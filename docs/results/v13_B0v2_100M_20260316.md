**Experiment:** `v6_100.0M_20260316_203510`
**Blueprint:** Fresh training (not continued from B0)

---

## Summary

Trained a new 100M-iteration blueprint (B0-v2) combining two v13 workstream features:
VR-MCCFR baseline variance reduction (WS2b) and the Zhang 2026 schedule (WS2c), plus
suit isomorphism (WS3). The result is a **regression** in exploitability: 1.3621 bb/100
vs B0's 1.2211 bb/100 (+0.141 bb/100 worse, 11.5% regression).

**Verdict:** Do not use B0-v2 for production. The combination of VR-MCCFR + Zhang schedule
is counterproductive at 100M iterations. Both features assume low-variance regret estimates,
which external sampling MCCFR does not provide — the same fundamental issue that caused the
PCFR+ negative result (v13 WS2a).

## Root Cause Analysis

The B0-v2 regression stems from the same fundamental issue as the PCFR+ negative result
(v13 WS2a): **techniques that assume low-variance estimates fail under external sampling
MCCFR, where consecutive estimates are independent high-variance Monte Carlo samples.**

### 1. VR-MCCFR cold-start problem

VR-MCCFR computes corrected values as:

```
v_vr(j) = sum_a sigma(a) * b(a) + v(j) - b(j)
```

where `b(a)` are per-action baselines. These baselines start at zero and are updated via
exponential moving average. During early iterations (before baselines warm up), the
correction term `v(j) - b(j)` with `b ≈ 0` adds the raw sample value as noise rather than
subtracting a correlated baseline. The "variance-reduced" estimates are actually
variance-*inflated* for the first ~500k iterations.

### 2. Zhang schedule amplifies early noise

The Zhang 2026 schedule applies time-varying discounting:

```
discount[t] = t^alpha / (t^alpha + c)    # alpha=2.0, c=1.0
weight[t] = (t / (t + 1))^beta           # beta=3.0
```

At early iterations, `discount[t]` is small, meaning old regrets are heavily discounted.
This is designed to "forget" suboptimal early play. However, when combined with VR-MCCFR's
cold-start noise, the schedule discounts the few iterations that *were* reasonably estimated
(before baselines diverge) and retains the noisy variance-inflated iterations.

### 3. Pattern recognition: the variance assumption family

This is the third v13 technique to fail for the same underlying reason:

| Technique | Assumption | Reality under external sampling |
|-----------|------------|--------------------------------|
| PCFR+ (WS2a) | `r_hat[t] = r[t-1]` is good predictor | Independent MC samples; prediction is uncorrelated noise |
| VR-MCCFR (WS2b) | Baselines reduce variance from iteration 0 | Baselines start at zero; add noise during warmup |
| Zhang schedule (WS2c) | Aggressive early discounting helps | Discounts the only reliable estimates, retains noisy ones |

All three techniques were validated in the literature for **full-tree CFR** (deterministic
regrets) or with **outcome sampling** with warm baselines. External sampling MCCFR with
fresh baselines violates these assumptions.

## Possible Fixes for B0-v3

### Option A: Delayed VR-MCCFR activation
Enable baselines only after a warmup phase (e.g., 500k iterations of standard CFR+). This
lets baselines accumulate meaningful estimates before they are used for variance reduction.
Implementation: add `--vr-warmup` flag to `train_gto.py`; run standard CFR+ for the warmup
period, then switch to VR-MCCFR.

### Option B: Exponential moving average baselines
Replace last-value baseline updates with an EMA: `b(a) = (1 - alpha) * b(a) + alpha * v(a)`.
This smooths out the per-iteration noise in baseline estimates, making the correction term
more stable. Trade-off: slower baseline adaptation but much lower variance during warmup.

### Option C: Isolate VR-MCCFR from Zhang schedule
Test each feature independently:
- **C1:** VR-MCCFR + standard linear CFR+ weighting (no Zhang)
- **C2:** Zhang schedule alone + no VR-MCCFR
- **C3:** Suit isomorphism alone + standard CFR+

This disentangles which feature(s) caused the regression. It is possible that suit
isomorphism alone is neutral and the regression is entirely from VR-MCCFR + Zhang.

### Option D: Pre-warm baselines from blueprint
Initialize VR-MCCFR baselines from the existing B0 blueprint's node values instead of
zero. This gives the baselines a reasonable starting point. Requires infrastructure to
extract per-node expected values from a trained strategy.
